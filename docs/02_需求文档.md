# AgentMemoryManager 产品需求文档（PRD）

**版本：** v0.1  
**日期：** 2026年5月  
**状态：** 草稿

---

## 一、背景与问题陈述

### 1.1 核心痛点

大语言模型（LLM）和基于 LLM 的智能体（Agent）在处理长文本或多轮对话时，存在严重的**上下文腐化**问题：

- **注意力稀释：** 相关信息随着对话轮次增加而被淹没，模型"遗忘"早期关键信息
- **成本膨胀：** 将全量历史塞入上下文窗口，token 消耗和 API 费用呈线性甚至指数增长
- **性能下降：** 研究表明，相关内容置于上下文中间位置时准确率下降超过 30%（"Lost in the Middle"，TACL 2024）
- **窗口上限：** 即便模型支持 128K token，实际有效信息处理能力远低于标称值

### 1.2 目标用户

| 用户类型 | 场景 | 核心需求 |
|---------|------|---------|
| **AI 应用开发者** | 开发聊天机器人、客服 Bot、个人助手 | 低接入成本、稳定 API、可观测性 |
| **Agent 框架开发者** | 构建多 Agent 系统（AutoGen、LangGraph 等） | 可插拔的记忆后端，支持跨 Agent 共享记忆 |
| **研究人员** | 评测 LLM 记忆能力 | 标准化评测接口、多策略对比 |
| **企业用户** | 长期运行的业务 Agent | 数据隔离、记忆持久化、审计日志 |

---

## 二、产品目标

### 2.1 核心目标

> 提供一个**可插拔的记忆管理组件**，使基于 LLM 的 Agent 在长对话和多轮任务中保持上下文连贯性，同时将 token 消耗和延迟控制在可接受范围内。

### 2.2 量化目标（v1.0 MVP）

| 指标 | 基准（无记忆管理） | 目标 |
|------|-----------------|------|
| P95 端到端延迟 | 9.87s（全量上下文） | < 2s |
| Token 消耗/会话（100 轮） | ~26,000 | < 4,000 |
| 多轮问答准确率（LoCoMo） | 72.9%（全上下文） | > 65%（目标 ≥ 68%） |
| 接入时间（新项目） | — | < 30 分钟 |

### 2.3 非目标（Out of Scope for v1.0）

- 不替代 LLM 本身，不提供推理能力
- 不提供 UI 界面（专注 SDK/API）
- 不支持多模态记忆（图像、音频）的存储（v2.0 规划）
- 不实现模型微调或 LoRA 适配

## 三、功能需求

### 3.1 记忆类型支持

#### FR-01：工作记忆（Working Memory）

- 管理当前对话窗口内的 token 预算
- 支持基于**重要性打分**的 token 级别压缩（LLMLingua 策略）
- 支持**滑动窗口**策略，保留最近 N 轮对话

#### FR-02：情节记忆（Episodic Memory）

- 自动将历史对话提取为**原子事实**（atomic facts）并持久化
- 支持基于**时间、重要性、相关性**三维打分的检索（Generative Agents 策略）
- 支持跨会话（cross-session）记忆持久化

#### FR-03：语义记忆（Semantic Memory）

- 维护实体（Entity）和关系（Relation）的结构化知识库
- 支持**知识图谱**存储后端（可选）
- 支持基于向量相似度的语义检索

#### FR-04：程序记忆（Procedural Memory）

- 存储成功执行的任务模板和工具使用模式
- 支持 Agent 跨任务复用已学习的操作流程

### 3.2 核心操作接口

#### FR-05：记忆写入（Memory Write）

- `add(content, session_id, metadata)` — 添加新记忆
- `update(memory_id, content)` — 更新已有记忆
- `delete(memory_id)` — 删除记忆

#### FR-06：记忆检索（Memory Retrieval）

- `search(query, session_id, top_k)` — 语义相似度检索
- `get_context(session_id, max_tokens)` — 获取注入上下文的格式化记忆
- `get_entities(session_id)` — 获取结构化实体记忆

#### FR-07：上下文注入（Context Injection）

- 提供 `build_prompt(base_prompt, session_id)` 接口，自动将相关记忆注入 prompt
- 支持自定义注入模板
- 注入的记忆需控制在指定 token 预算内

#### FR-08：记忆压缩（Memory Compression）

- 支持对历史记忆进行**自动摘要压缩**（Summarization）
- 支持**反思触发**（Reflection Trigger）：当重要性积分超过阈值时自动合成高层洞察
- 支持手动触发 `compress(session_id)`

### 3.3 存储后端

#### FR-09：多后端支持

| 后端 | 用途 | 必选/可选 |
|------|------|---------|
| 内存（In-Memory） | 开发/测试，无持久化 | **必选** |
| SQLite | 轻量本地持久化 | **必选** |
| PostgreSQL + pgvector | 生产级向量存储 | 可选 |
| Qdrant / Chroma / Weaviate | 专用向量数据库 | 可选 |
| Neo4j | 知识图谱后端 | 可选 |
| Redis | 高速缓存层 | 可选 |

#### FR-10：后端接口抽象

- 定义统一的 `MemoryBackend` 抽象接口，支持用户自定义实现

### 3.4 策略模块

#### FR-11：可插拔记忆策略

支持以下内置策略，用户可通过配置文件选择或组合：

| 策略 ID | 策略名称 | 参考来源 |
|---------|---------|---------|
| `sliding_window` | 滑动窗口 | LangChain ConversationBufferWindowMemory |
| `summarize` | 历史摘要 | LangChain ConversationSummaryMemory |
| `atomic_facts` | 原子事实提取 | Mem0 |
| `reflection` | 反思机制 | Generative Agents |
| `zettelkasten` | 动态笔记链接 | A-MEM |
| `compression` | Token 压缩 | LLMLingua |

#### FR-12：策略组合（Pipeline）

- 支持将多个策略串联为 Pipeline
- 例如：`summarize → atomic_facts → reflection`

### 3.5 LLM 提供商支持

#### FR-13：多 LLM 支持

- 支持 Anthropic Claude（claude-sonnet-4-6、claude-opus-4-7 等）
- 支持 OpenAI GPT-4o 系列
- 支持本地模型（通过 Ollama / LM Studio）
- 支持 LiteLLM 统一接入层

### 3.6 集成与互操作性

#### FR-14：框架集成

- 提供 **LangChain** 兼容的 Memory 类（`BaseMemory` 子类）
- 提供 **LlamaIndex** 兼容的记忆模块
- 提供 Python SDK（主要）
- 提供 REST API 服务模式（可选部署）

## 四、非功能需求

### 4.1 性能

- NFR-01：`search()` 接口 P99 延迟 < 500ms（不含 LLM 调用）
- NFR-02：`add()` 接口 P99 延迟 < 200ms（不含 LLM 调用）
- NFR-03：单会话支持最多 10,000 轮对话，不降级

### 4.2 可靠性

- NFR-04：LLM API 失败时，记忆系统降级为滑动窗口策略，不中断主流程
- NFR-05：存储层异常时，支持 fallback 到内存模式

### 4.3 可观测性

- NFR-06：提供结构化日志（JSON），记录每次记忆写入/检索/压缩事件
- NFR-07：暴露 Prometheus 指标（记忆命中率、压缩比、延迟分布）
- NFR-08：支持 Trace ID 传播，与调用方链路追踪集成

### 4.4 安全与隐私

- NFR-09：支持按 `user_id` 或 `session_id` 进行**数据隔离**
- NFR-10：支持静态加密（at-rest encryption）选项
- NFR-11：提供记忆**完全删除**（GDPR right to be forgotten）接口

### 4.5 可扩展性

- NFR-12：支持水平扩展（向量数据库 + 无状态服务层）
- NFR-13：支持多租户（multi-tenant）场景

## 五、用户故事

### 5.1 核心用户故事

**US-01：开发者快速接入**
> 作为一名 AI 应用开发者，我希望在 30 分钟内将 AgentMemoryManager 集成到我现有的 LangChain 应用中，不需要修改超过 10 行代码。

**US-02：长对话不遗忘**
> 作为一名客服 Bot 用户，我希望 Bot 在 100 轮对话后仍然记得我在第 1 轮提到的姓名和问题背景，而不是每次都需要重新介绍自己。

**US-03：成本控制**
> 作为一名 AI 应用运营者，我希望系统在保持回答质量的前提下，将每次 API 调用的 token 消耗控制在合理范围，不随对话轮次线性增长。

**US-04：跨会话记忆**
> 作为一名个人助手用户，我希望明天新开一个会话时，助手还记得我上周提到的项目进度和偏好设置。

**US-05：可审计**
> 作为一名企业 IT 管理员，我希望查看某个 Agent 会话的完整记忆演化历史，以便排查问题或满足合规要求。

## 六、约束与假设

### 6.1 约束

- 初始版本（v1.0）仅支持 Python 3.10+
- 不假设用户有特定的 LLM 提供商；必须支持至少 2 个主流提供商
- 必须支持离线（airgapped）部署模式（使用本地 LLM）

### 6.2 假设

- 用户已有可用的 LLM API 访问权限
- 对话内容以文本为主（v1.0 不涉及多模态）
- 用户能够接受轻微的准确率损失（<10%）以换取显著的成本和延迟优化
